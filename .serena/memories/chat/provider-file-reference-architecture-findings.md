Long-term attachment fix planning for Issue #253: current architecture stores local file paths in MessageV2.files and every provider request transform in ChatRepositoryImpl inlines attachments as base64 during request construction, which still causes large background-heap allocations on retry. Official provider capabilities gathered during planning: OpenAI Responses supports input_image.file_id alongside image_url and has Uploads/Files APIs; Anthropic has a beta Files API with reusable file_id references for Messages; Gemini Files API supports uploaded files referenced by file_uri and docs explicitly recommend Files API over inline bytes for larger/reused images, noting inline data contributes to a 20MB total request limit. User-selected product decisions for redesign: use provider-backed file references for OpenAI/Anthropic/Google, keep a smaller safe inline fallback for Groq/Ollama/OpenRouter/CUSTOM, upload provider files on Send (not on attach), and persist provider refs in Room so retry/edit can reuse them across app restarts. Likely implementation requires replacing MessageV2.files: List<String> with a richer persisted attachment model plus DB migration, while keeping local file info for UI thumbnails and storing per-provider remote refs/status metadata.
